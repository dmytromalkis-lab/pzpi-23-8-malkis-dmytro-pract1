3 ОПИС ВИКОНАНОЇ РОБОТИ 
 3.1 Загальні відомості про систему
Назва системи: YouTube.
Розробник: Google.
Призначення системи:
Глобальна платформа для обміну, зберігання та потокового відтворення відеоконтенту. Система забезпечує користувачам можливість перегляду, завантаження, коментування та поширення відео через мережу Інтернет. Цільова аудиторія охоплює мільярди користувачів у всьому світі.
Основні функції:
* реєстрація та автентифікація користувачів;
* завантаження та зберігання відеофайлів;
* транскодування відео у різні формати та якості;
* адаптивний стрімінг відеоконтенту;
* система персоналізованих рекомендацій;
* пошук та фільтрація відео;
* система коментарів, підписок та вподобань;
* проведення прямих трансляцій;
* збір статистики та аналітики переглядів.
3.2 Архітектурний стиль системи
Система YouTube використовує розподілену мікросервісну архітектуру у поєднанні з хмарною інфраструктурою та глобальною мережею серверів доставки контенту (CDN).
Обґрунтування вибору:
Використання мікросервісної архітектури обумовлене необхідністю забезпечення високої масштабованості, відмовостійкості та стабільної роботи системи при обслуговуванні мільйонів користувачів одночасно. Поділ системи на окремі незалежні сервіси дозволяє розробникам паралельно оновлювати та вдосконалювати різні компоненти платформи без зупинки роботи всієї системи.


Кожен мікросервіс відповідає за окрему функцію, наприклад:
*  обробку відео;
* систему рекомендацій;
* авторизацію користувачів;
* пошук контенту;
* аналітику та статистику;
* потокову передачу відео.
Хмарна інфраструктура та розподілені сервери дозволяють автоматично масштабувати ресурси залежно від навантаження. Це особливо важливо під час пікової активності користувачів або проведення прямих трансляцій.
Додатково використання CDN забезпечує швидку доставку відеоконтенту користувачам із найближчих серверів, що зменшує затримки та підвищує швидкість відтворення відео.
3.3 Основні компоненти системи
Опис підсистем YouTube:
* Інтерфейс користувача:
Клієнтська частина системи представлена застосунками для Smart TV, мобільних пристроїв (Android, iOS), ігрових консолей та веб-браузерів. Інтерфейс адаптується під різні типи екранів і пристроїв, забезпечуючи зручну навігацію, пошук відео, перегляд контенту та взаємодію з користувачем (лайки, коментарі, підписки).
* Серверна частина:
Вхідні запити від клієнтів обробляються через API Gateway, який виконує маршрутизацію запитів до відповідних сервісів. Він також забезпечує базовий рівень безпеки, автентифікацію та балансування навантаження між сервісами.
* Рівень бізнес-логіки:
Система побудована на основі великої кількості незалежних мікросервісів, які розгорнуті у хмарній інфраструктурі Google Cloud. До основних сервісів належать:
* сервіс авторизації та керування акаунтами;
* сервіс профілів користувачів;
* рушій персоналізованих рекомендацій;
* сервіс обробки відео та транскодування;
* білінгові та аналітичні сервіси.
* Рівень зберігання даних:
Для ефективної роботи з великими обсягами інформації використовується гетерогенна система баз даних:
* NoSQL-бази (наприклад, Cassandra) — для зберігання великих потоків даних та логів;
* кеш-системи (Redis, Memcached) — для пришвидшення доступу до часто використовуваних даних;
* реляційні бази даних (MySQL) — для транзакційних операцій і структурованих даних.
* Допоміжні сервіси:
Для забезпечення стабільної та масштабованої роботи використовуються додаткові компоненти:
* глобальна CDN-система (YouTube Open Connect) для швидкої доставки відеоконтенту;
* система Service Discovery (наприклад, Eureka або аналогічні рішення) для автоматичного виявлення сервісів у кластері;
* інструменти Chaos Engineering для тестування стійкості системи до збоїв та аварійних ситуацій.
У сукупності ці компоненти забезпечують високу продуктивність, масштабованість та надійність всієї платформи YouTube.
3.4 Взаємодія компонентів
Взаємодія між компонентами YouTube побудована на принципах розподіленої архітектури, де кожен етап обробки запиту виконується окремими сервісами, що дозволяє забезпечити високу продуктивність та масштабованість системи.
Коли користувач взаємодіє з платформою (наприклад, відкриває сторінку або запускає відео), процес обробки запиту проходить кілька послідовних етапів:
   1. Клієнтський запит
 Клієнтський пристрій (веб-браузер, мобільний застосунок або Smart TV) надсилає HTTP/HTTPS-запит до системи.
   2. Балансування навантаження
 Запит спочатку потрапляє до балансувальника навантаження (наприклад, AWS Elastic Load Balancing), який розподіляє трафік між доступними серверами, забезпечуючи рівномірне навантаження та відмовостійкість.
   3. API Gateway
 Далі запит передається до API Gateway, який виконує:
      * маршрутизацію запитів до відповідних сервісів;
      * перевірку автентифікації користувача;
      * базову агрегацію даних;
      * контроль доступу та обмеження запитів.
      4. Service Discovery (Eureka)
API Gateway звертається до системи Service Discovery (наприклад, Eureka), щоб отримати актуальну інформацію про розташування потрібного мікросервісу (IP-адресу або endpoint). Це дозволяє системі динамічно масштабувати сервіси без жорсткої прив’язки до конкретних серверів.
      5. Обробка запиту мікросервісом
 Вибраний мікросервіс (наприклад, сервіс рекомендацій або пошуку) обробляє запит. У процесі він може:
         * звернутися до кешу (Redis або Memcached) для швидкого отримання даних;
         * у разі відсутності даних у кеші — виконати запит до бази даних (наприклад, Cassandra або іншої NoSQL-системи);
         * застосувати бізнес-логіку (наприклад, алгоритми рекомендацій або фільтрації контенту).
         6. Формування відповіді
Після обробки запиту мікросервіс повертає результат у форматі JSON до API Gateway, який передає його клієнту. Клієнтський застосунок відображає отримані дані у вигляді інтерфейсу користувача.
         7. Відтворення відео (CDN шлях)
Коли користувач натискає кнопку "Play", система не передає відео через бізнес-логіку або API Gateway. Замість цього:
            * клієнт отримує URL на медіа-файл;
            * запит на завантаження відео спрямовується безпосередньо до найближчого вузла CDN (YouTube Open Connect);
            * CDN доставляє відео з географічно найближчого сервера, мінімізуючи затримки та навантаження на центральні системи.
            8. Оптимізації та додаткові механізми
 У процесі взаємодії також використовуються:
               * кешування результатів запитів для зменшення навантаження;
               * асинхронна обробка деяких операцій (наприклад, аналітики переглядів);
               * черги повідомлень для обробки подій у фоновому режимі.
У результаті така схема взаємодії дозволяє YouTube ефективно обробляти мільйони запитів одночасно, забезпечуючи стабільну роботу та швидке відтворення відео навіть при високому навантаженні.
3.5 Масштабованість та надійність
Система YouTube побудована таким чином, щоб ефективно працювати під надзвичайно високими навантаженнями та забезпечувати безперервний доступ до сервісу для мільярдів користувачів у всьому світі. Для цього використовуються комплексні механізми масштабування та забезпечення надійності.
               * Механізми балансування навантаження:
 Для розподілу вхідного трафіку застосовуються як зовнішні, так і внутрішні балансувальники. Зовнішній рівень реалізується через AWS Elastic Load Balancing, який розподіляє запити між доступними вузлами системи. На рівні мікросервісів використовується внутрішнє балансування (наприклад, Ribbon або аналогічні рішення), що дозволяє рівномірно розподіляти навантаження між екземплярами сервісів та уникати перевантаження окремих компонентів.
               * Відмовостійкість:
 Для забезпечення стабільної роботи системи застосовується патерн Circuit Breaker (наприклад, Hystrix або його сучасні аналоги). Він дозволяє ізолювати несправні мікросервіси та запобігати каскадному поширенню збоїв по всій системі.
Додатково використовується підхід Chaos Engineering, зокрема інструменти на кшталт Chaos Monkey, які навмисно моделюють збої в системі для перевірки її стійкості до реальних аварійних ситуацій.
               * Кешування:
 Для зменшення затримок та зниження навантаження на бази даних використовується багаторівневе кешування (наприклад, EVCache). У кеші зберігаються часто запитувані дані, зокрема профілі користувачів, рекомендації та інші персоналізовані дані, що дозволяє значно прискорити обробку запитів.
               * Реплікація даних:
 Усі критичні дані дублюються між кількома географічно розподіленими зонами доступності (AWS Availability Zones). Це забезпечує високу доступність системи навіть у разі виходу з ладу окремого дата-центру та мінімізує ризик втрати даних.
               * Хмарна інфраструктура:
 Система повністю базується на еластичній хмарній інфраструктурі Amazon Web Services. Використання сервісів EC2, S3 та інших дозволяє динамічно виділяти обчислювальні ресурси залежно від поточного навантаження, забезпечуючи гнучкість, масштабованість та економічну ефективність.
У сукупності ці механізми роблять систему YouTube високонадійною, стійкою до збоїв та здатною підтримувати стабільну роботу навіть при різких піках навантаження.


3.6 Приклади програмної реалізації
Для демонстрації принципів мікросервісної архітектури та механізмів забезпечення відмовостійкості в системі YouTube було створено навчальний приклад програмної реалізації API Gateway у поєднанні з патерном Circuit Breaker. Особлива увага приділяється забезпеченню безперервної роботи сервісу рекомендацій відео навіть у випадку часткових збоїв системи.
Запит до ШІ:
«Згенеруй базовий приклад роботи API Gateway разом із Circuit Breaker (на прикладі Java/Spring Boot), який демонструє обробку збоїв сервісу рекомендацій YouTube та повернення резервного результату».
1  import org.springframework.beans.factory.annotation.Autowired;
2  import org.springframework.web.bind.annotation.*;
3  import org.springframework.web.client.RestTemplate;
4  import io.github.resilience4j.circuitbreaker.annotation.CircuitBreaker;
5  import java.util.Arrays;
6  import java.util.List;
7
8  @RestController
9  public class VideoRecommendationService {
10
11     @Autowired
12     private RestTemplate restTemplate;
13
14     // Запит до зовнішнього сервісу рекомендацій відео
15     @GetMapping("/videos/{userId}")
16     @CircuitBreaker(name = "youtubeRecommendationService",
17                     fallbackMethod = "getFallbackVideos")
18     public List<String> getRecommendedVideos(@PathVariable String userId) {
19
20         // Імітація виклику зовнішнього мікросервісу рекомендацій
21         return restTemplate.getForObject(
22             "http://recommendation-service/api/videos/" + userId,
23             List.class
24         );
25     }
26
27     // Резервний метод при відмові recommendation-service
28     public List<String> getFallbackVideos(String userId, Exception e) {
29
30         // Повернення стандартного списку популярних відео
31         return Arrays.asList(
32             "Популярне відео 1",
33             "Популярне відео 2",
34             "Популярне відео 3"
35         );
36     }
37 }


Наведений клас VideoRecommendationService виконує мережевий виклик до зовнішнього сервісу рекомендацій відео. За допомогою анотації @CircuitBreaker (рядки 16–17) система автоматично контролює стан зовнішнього сервісу та реагує на його можливі збої.
У випадку недоступності сервісу рекомендацій, перевищення часу очікування або виникнення помилки мережі виняток безпечно перехоплюється механізмом Circuit Breaker. Замість повернення критичної помилки HTTP 500 система автоматично перенаправляє виконання до резервного методу getFallbackVideos() (рядок 28).
Fallback-метод повертає користувачу базовий список популярних відео, які можуть бути попередньо закешованими або сформованими за замовчуванням. Таким чином реалізується принцип «graceful degradation» — система продовжує частково функціонувати навіть у випадку збоїв окремих мікросервісів.
Подібний підхід широко використовується у високонавантажених системах на зразок YouTube, оскільки дозволяє:
                  * запобігати каскадним збоям;
                  * зменшувати навантаження на несправні сервіси;
                  * підтримувати стабільну роботу платформи;
                  * покращувати користувацький досвід навіть при часткових відмовах інфраструктури.
Завдяки використанню Circuit Breaker система YouTube може забезпечувати безперервний доступ до контенту та зберігати працездатність критично важливих компонентів навіть під час високих навантажень або аварійних ситуацій.

3.9 Використання ШІ
Назва інструменту та версія: Chat-GPT-5.5
Мета використання:
 Штучний інтелект використовувався для аналізу та узагальнення технічної інформації щодо архітектури платформи YouTube, принципів побудови розподілених мікросервісних систем, механізмів масштабування та забезпечення відмовостійкості. Окремо ШІ застосовувався для генерації прикладу програмного коду на Java/Spring Boot, який демонструє використання патерна Circuit Breaker у взаємодії між мікросервісами.

1  import org.springframework.beans.factory.annotation.Autowired;
2  import org.springframework.web.bind.annotation.*;
3  import org.springframework.web.client.RestTemplate;
4  import io.github.resilience4j.circuitbreaker.annotation.CircuitBreaker;
5  import java.util.Arrays;
6  import java.util.List;
7
8  @RestController
9  public class VideoRecommendationService {
10
11     @Autowired
12     private RestTemplate restTemplate;
13
14     // Запит до зовнішнього сервісу рекомендацій відео
15     @GetMapping("/videos/{userId}")
16     @CircuitBreaker(name = "youtubeRecommendationService",
17                     fallbackMethod = "getFallbackVideos")
18     public List<String> getRecommendedVideos(@PathVariable String userId) {
19
20         // Імітація виклику зовнішнього мікросервісу рекомендацій
21         return restTemplate.getForObject(
22             "http://recommendation-service/api/videos/" + userId,
23             List.class
24         );
25     }
26
27     // Резервний метод при відмові recommendation-service
28     public List<String> getFallbackVideos(String userId, Exception e) {
29
30         // Повернення стандартного списку популярних відео
31         return Arrays.asList(
32             "Популярне відео 1",
33             "Популярне відео 2",
34             "Популярне відео 3"
35         );
36     }
37 }
