3 ОПИС ВИКОНАНОЇ РОБОТИ 
3.1 Загальні відомості про шаблони (патерни) проєктування
Шаблони проєктування (design patterns) являють собою типові та багаторазово перевірені підходи до вирішення поширених задач під час розробки об’єктно-орієнтованих програмних систем. Вони не виступають готовими програмними модулями чи фрагментами коду, а описують загальні принципи організації структури та взаємодії компонентів системи.
Найбільш відомою є класифікація, запропонована «Бандою чотирьох» (GoF), у межах якої виділяють 23 базові патерни. Усі вони поділяються на три основні категорії:
* твірні — відповідають за ефективне створення об’єктів;
* структурні — визначають способи формування складної архітектури системи;
* поведінкові — описують механізми взаємодії між об’єктами та алгоритми їхньої співпраці.
Використання шаблонів проєктування дозволяє підвищити гнучкість програмного забезпечення, спростити його підтримку та модернізацію, а також забезпечити кращу масштабованість системи під час її подальшого розвитку.
3.2 Опис обраного патерна
* Назва патерна: Абстрактна фабрика (Abstract Factory)
* Тип шаблона (патерна): твірний
* Основне призначення: забезпечення інтерфейсу для створення сімейств взаємопов’язаних або залежних об’єктів без необхідності вказувати їх конкретні класи. Патерн дозволяє створювати групи сумісних об’єктів, приховуючи деталі їх реалізації від клієнтського коду.
* Передумови застосування: патерн доцільно використовувати у випадках, коли система повинна підтримувати декілька варіантів або платформ реалізації; коли необхідно забезпечити сумісність між створюваними об’єктами; а також для ізоляції процесу створення об’єктів від основної бізнес-логіки застосунку.
* Переваги: забезпечення слабкого зв’язку між клієнтським кодом і конкретними класами; спрощення заміни сімейств об’єктів; дотримання принципу відкритості/закритості (OCP), що дозволяє додавати нові фабрики без зміни існуючого коду; підвищення масштабованості та підтримуваності системи.
* Недоліки: збільшення складності архітектури через необхідність створення додаткових інтерфейсів і класів фабрик; ускладнення структури проєкту при додаванні нових типів продуктів, оскільки потрібно змінювати всі пов’язані фабрики.
3.3 Структура шаблона (патерна)
* Опис структурних частин:
* Абстрактна фабрика (Abstract Factory): визначає загальний інтерфейс для створення сімейств взаємопов’язаних об’єктів без прив’язки до їх конкретних реалізацій.
* Конкретна фабрика (Concrete Factory): реалізує інтерфейс Abstract Factory та відповідає за створення конкретних варіантів продуктів, які належать до певного сімейства.
* Абстрактний продукт (Abstract Product): оголошує спільний інтерфейс для всіх типів продуктів, що створюються фабрикою.
* Конкретний продукт (Concrete Product): реалізує інтерфейс Abstract Product та представляє конкретний об’єкт певного сімейства продуктів.
* Клієнт (Client): використовує лише інтерфейси Abstract Factory та Abstract Product, не взаємодіючи напряму з конкретними класами продуктів і фабрик.
* Опис взаємодії об’єктів: Клієнт звертається до Abstract Factory через визначений інтерфейс для створення необхідних об’єктів. Конкретна фабрика створює відповідні конкретні продукти певного сімейства та повертає їх клієнту. Надалі Клієнт працює з отриманими об’єктами через абстрактні інтерфейси продуктів, не залежачи від деталей їх реалізації. Це дозволяє легко змінювати сімейства продуктів без модифікації основної логіки системи.
3.4 Реалізація шаблона (патерна)
* Опис реалізації мовою програмування C#
* Основні етапи:
* Створення базових інтерфейсів: розробка абстрактного інтерфейсу IAbstractFactory, який визначає методи створення взаємопов’язаних об’єктів, а також створення інтерфейсів абстрактних продуктів (IProductA, IProductB), що описують спільну функціональність майбутніх компонентів системи.
* Реалізація конкретних класів: створення класів конкретних фабрик (ConcreteFactory1, ConcreteFactory2), які реалізують IAbstractFactory та відповідають за створення певного сімейства продуктів. Для кожної фабрики створюються відповідні класи конкретних продуктів (ConcreteProductA1, ConcreteProductB1 тощо), які реалізують інтерфейси абстрактних продуктів.
* Демонстрація використання: у методі Main створюється екземпляр конкретної фабрики, через яку клієнтський код отримує необхідні об’єкти. Надалі робота виконується виключно через абстрактні інтерфейси, що дозволяє легко замінювати сімейства продуктів без зміни логіки клієнтської частини програми.

using System;


// 1. Абстрактні продукти (Abstract Products)
public interface IButton
{
    void Render();
}


public interface ICheckbox
{
    void Check();
}


// 2. Конкретні продукти для Windows
public class WindowsButton : IButton
{
    public void Render()
    {
        Console.WriteLine("WindowsButton: Відображення кнопки Windows");
    }
}


public class WindowsCheckbox : ICheckbox
{
    public void Check()
    {
        Console.WriteLine("WindowsCheckbox: Позначення Windows Checkbox");
    }
}


// 3. Конкретні продукти для MacOS
public class MacButton : IButton
{
    public void Render()
    {
        Console.WriteLine("MacButton: Відображення кнопки MacOS");
    }
}


public class MacCheckbox : ICheckbox
{
    public void Check()
    {
        Console.WriteLine("MacCheckbox: Позначення MacOS Checkbox");
    }
}


// 4. Абстрактна фабрика (Abstract Factory)
public interface IGUIFactory
{
    IButton CreateButton();
    ICheckbox CreateCheckbox();
}


// 5. Конкретні фабрики (Concrete Factories)
public class WindowsFactory : IGUIFactory
{
    public IButton CreateButton()
    {
        return new WindowsButton();
    }


    public ICheckbox CreateCheckbox()
    {
        return new WindowsCheckbox();
    }
}


public class MacFactory : IGUIFactory
{
    public IButton CreateButton()
    {
        return new MacButton();
    }


    public ICheckbox CreateCheckbox()
    {
        return new MacCheckbox();
    }
}


// 6. Клієнтський код (Client)
class Program
{
    static void Main()
    {
        // Вибір фабрики
        IGUIFactory factory;


        string systemType = "Windows";


        if (systemType == "Windows")
        {
            Console.WriteLine("Створення компонентів Windows...");
            factory = new WindowsFactory();
        }
        else
        {
            Console.WriteLine("Створення компонентів MacOS...");
            factory = new MacFactory();
        }


        // Створення продуктів через фабрику
        IButton button = factory.CreateButton();
        ICheckbox checkbox = factory.CreateCheckbox();


        // Використання створених об'єктів
        button.Render();
        checkbox.Check();
    }
}

* Опиши структуру шаблона проєктування «Abstract Factory», вказавши основні структурні елементи (Abstract Factory, Concrete Factory, Abstract Product, Concrete Product, Client) та принцип їх взаємодії між собою. 
* Розроби приклад програмного коду мовою C#, який демонструє класичну реалізацію патерна «Abstract Factory» на прикладі створення графічних компонентів для різних операційних систем (Windows та MacOS). 
* Наведи декілька прикладів із сучасної програмної інженерії, де застосування патерна «Abstract Factory» є найбільш доцільним, наприклад створення кросплатформених інтерфейсів або робота з різними типами баз даних.
